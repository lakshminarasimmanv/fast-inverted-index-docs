System Architecture

Overview​

Fast-Inverted-Index is a high-performance search engine library implemented in Rust with Python bindings. It provides efficient term-to-document mapping for full-text search capabilities while optimizing for both memory usage and query performance.

Core Components​

The system is organized into several key components, each responsible for a specific aspect of the search functionality:

1. Dictionary​

The dictionary component manages term storage and lookup using a hybrid approach:

  • Hybrid Dictionary: Multi-tiered storage optimized for different access patterns
    • Hot terms: Hash-based structure for O(1) lookup of frequently accessed terms
    • Medium terms: Radix Trie for O(k) lookup with prefix compression
    • Cold terms: Simple hash map for infrequently accessed terms
  • Automatic promotion/demotion of terms between tiers based on access patterns
  • Memory optimization through string interning
  • Fast prefix matching for autocomplete/suggestions

2. Posting Lists​

Posting lists store document occurrences for each term with:

  • Skip list optimization for faster list intersections
  • Variable-length integer (varint) encoding for compression
  • Position tracking for potential phrase queries

3. Storage Layer​

The storage layer provides persistence capabilities through:

  • RocksDB integration for reliable on-disk storage
  • Column families for organizing different data types
  • Serialization/deserialization with efficient binary formats

4. Query Engine​

The query engine handles search operations with support for:

  • Boolean operators (AND, OR)
  • Term frequency scoring
  • BM25 and TF-IDF ranking algorithms

5. Cache​

The caching system improves performance through:

  • LRU (Least Recently Used) caching strategy
  • Configurable TTL (Time-To-Live)
  • Thread-safe implementation

Data Flow​

  1. Indexing Flow:

    • Documents are added through the indexing API
    • Text is tokenized into terms
    • Terms are added to the dictionary
    • Postings are created and added to posting lists
    • Data is persisted to storage

  2. Query Flow:

    • Query is parsed into terms and operators
    • Terms are looked up in the dictionary
    • Posting lists are retrieved (from cache if available)
    • Lists are intersected or united based on query operators
    • Results are ranked using the specified algorithm
    • Document metadata is retrieved and returned

Design Decisions​

Performance Optimizations​

  • Hybrid Dictionary: Multi-tiered term storage with adaptive promotion/demotion optimizes for Zipfian term distribution in real-world search.
  • Skip Lists: Block-level skip pointers reduce the number of comparisons needed during list intersections, improving query performance.
  • Varint Encoding: Variable-length integer encoding reduces storage requirements while maintaining good decompression performance.
  • LRU Caching: Frequently accessed posting lists are kept in memory to avoid disk access.
  • String Interning: Deduplicates string storage for terms, reducing memory footprint.
  • Radix Trie: Efficient prefix-based term storage enables fast suggestion generation and memory savings.

Concurrency​

  • Thread-safe cache implementation
  • Immutable references where possible
  • Transaction-like operations in the storage layer

Memory Management​

  • Custom allocators for large collection types
  • Efficient binary serialization formats
  • Lazy loading of posting lists

Implementation Details​

Component Interactions​

Components interact through well-defined interfaces:

  • The dictionary provides term-to-posting-list mapping
  • The query engine uses the dictionary to resolve terms
  • The storage layer provides persistence for all components
  • The cache intercepts storage access for frequently used data

Error Handling Strategy​

The library uses a custom error type hierarchy with:

  • Specific error variants for different failure cases
  • Informative error messages
  • Proper error propagation through the Result type
  • Conversion to Python exceptions in the bindings

Technology Choices​

  • Rust: For performance, memory safety, and concurrency
  • RocksDB: For efficient key-value storage
  • PyO3: For high-quality Python bindings
  • Varint: For efficient integer compression
