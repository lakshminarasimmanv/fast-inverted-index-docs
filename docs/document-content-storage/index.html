Document Content Storage in Fast Inverted Index

This document explains how document content is handled in the Fast Inverted Index library and provides best practices for content retrieval.

How Content is Handled​

When adding documents to the Fast Inverted Index, the content is indexed but not stored directly in the index. This distinction is important to understand:

  1. Indexing: The content is analyzed, tokenized, and added to the inverted index for searching.
  2. Storage: The content itself is not stored in a way that can be directly retrieved.

This design choice has several implications:

  • The index is more efficient and compact
  • Searches against content work as expected
  • Document metadata is preserved and retrievable
  • The original content must be stored separately if retrieval is needed

Retrieving Documents​

When you retrieve a document using get_document(doc_id), you get a dictionary containing:

{
  "id": 1,
  "title": "Document Title",
  "author": "Author Name",
  "tags": ["tag1", "tag2"],
  "language": "en",
  "created_at": 1234567890,
  "updated_at": 1234567890,
  "metadata": {
    "word_count": 150,
    "reading_time": 45,
    "byte_length": 750
  }
}

Notice that the full document content is not included in this structure.

Best Practices for Content Retrieval​

1. Store Content Externally​

Store the original document content in a separate storage system:

# Content store could be a database, file system, or in-memory dictionary
content_store = {}

# When adding documents
doc_id = 1
content = "This is the document content..."
fields = {"title": "Document Title", "author": "Author Name"}

# Store in index
index.add_document(doc_id, content, fields)

# Store content separately
content_store[doc_id] = content

# When retrieving documents
search_results = index.search("query")
for doc_id, score in search_results:
    doc_metadata = index.get_document(doc_id)
    full_content = content_store[doc_id]
    
    print(f"Title: {doc_metadata['title']}")
    print(f"Score: {score}")
    print(f"Content: {full_content}")

2. Include Content in Fields​

If you need the content retrievable directly from the index, include it in the fields dictionary:

doc_id = 1
content = "This is the document content..."
fields = {
    "title": "Document Title",
    "author": "Author Name",
    "content": content  # Duplicate content in fields
}

# Store in index with content both as primary content and in fields
index.add_document(doc_id, content, fields)

# When retrieving
doc = index.get_document(doc_id)
if "content" in doc:
    print(f"Content: {doc['content']}")

However, in our testing, we found that the content field is not reliably preserved when using this approach.

3. Use a Storage-Enabled Index Implementation​

Some search engines (like Elasticsearch) provide both indexing and storage capabilities. If you need full content retrieval, consider:

  1. Using a different search engine implementation
  2. Building a storage layer on top of Fast Inverted Index
  3. Implementing a document store integration

Example Implementation​

A typical implementation might look like this:

class SearchSystem:
    def __init__(self):
        self.index = Index(in_memory=True)
        self.document_store = {}  # Could be a database
    
    def add_document(self, doc_id, content, metadata):
        # Index for searching
        self.index.add_document(doc_id, content, metadata)
        # Store for retrieval
        self.document_store[doc_id] = {
            "content": content,
            "metadata": metadata
        }
    
    def search(self, query, limit=10):
        # Search the index
        results = self.index.search(query, limit=limit)
        
        # Enrich results with content
        enriched_results = []
        for doc_id, score in results:
            metadata = self.index.get_document(doc_id)
            content = self.document_store[doc_id]["content"]
            
            enriched_results.append({
                "id": doc_id,
                "score": score,
                "metadata": metadata,
                "content": content
            })
            
        return enriched_results

Conclusion​

The Fast Inverted Index library is primarily designed for efficient searching rather than document storage. For a complete search system, you'll need to implement a separate storage layer to preserve and retrieve document content.

This separation of concerns is common in search system design, allowing each component to be optimized for its specific purpose: the index for fast searching and the document store for content retrieval.
