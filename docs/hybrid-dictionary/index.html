Hybrid Dictionary Implementation

Overview​

The Hybrid Dictionary is an advanced data structure implemented in the Fast-Inverted-Index library to optimize term storage and lookup. It combines multiple specialized data structures to achieve an optimal balance between memory efficiency, lookup speed, and feature support, addressing the performance bottlenecks observed in large-scale search operations.

This document details the implementation, design considerations, and performance characteristics of the Hybrid Dictionary.

Motivation​

In real-world search applications, term distributions typically follow a power law (Zipf distribution):

  • A small number of terms (10-20%) account for the majority of lookups (80-90%)
  • A large number of rare terms consume memory but have infrequent lookups

The original implementation used a RadixTrie for all terms, which provided good memory efficiency but didn't optimize for this access pattern. The Hybrid Dictionary addresses this by using different storage strategies for terms based on their access frequency.

Design Philosophy​

The Hybrid Dictionary follows these key design principles:

  1. Frequency-Based Tiering: Store terms in different data structures based on their access frequency
  2. Memory Efficiency: Use advanced techniques like string interning to minimize memory overhead
  3. Adaptive Behavior: Dynamically move terms between tiers based on observed access patterns
  4. Minimal API Changes: Maintain backward compatibility with existing codebase
  5. Thread Safety: Support concurrent access for multi-threaded applications

Architecture​

The Hybrid Dictionary consists of three primary tiers:

1. Hot Terms Layer​

  • Purpose: Store frequently accessed terms with minimal lookup overhead
  • Implementation: Hash-based structure (HashMap<InternedString, TermEntry<V>>)
  • Characteristics:
    • O(1) lookup time
    • Uses more memory per term
    • Optimized for speed over space efficiency
    • Holds top N terms (configurable, default 10,000)

2. Medium Terms Layer (Trie Terms)​

  • Purpose: Store medium-frequency terms with balanced performance
  • Implementation: Radix Trie (RadixTrie<TermEntry<V>>)
  • Characteristics:
    • O(k) lookup time where k is term length
    • Good space efficiency through prefix compression
    • Supports efficient prefix-based operations
    • Well-suited for suggestion/autocomplete features

3. Cold Terms Layer​

  • Purpose: Store rare terms with minimal memory overhead
  • Implementation: Standard hash map (HashMap<String, TermEntry<V>>)
  • Characteristics:
    • Simple implementation
    • Moderate memory usage
    • No specialized prefix operations needed for rare terms

Auxiliary Components​

String Interner​

A critical optimization is the String Interner, which:

  • Deduplicates string storage for terms
  • Provides reference semantics for efficient comparison
  • Reduces memory footprint for repeated strings
  • Enables O(1) string equality checking

Term Metadata​

Each term is wrapped in a TermEntry structure that tracks:

  • The associated value (term ID)
  • Access frequency counter
  • Last access timestamp
  • Additional serialization/deserialization support

Adaptive Behavior​

The Hybrid Dictionary implements a promotion/demotion system to adapt to actual usage patterns:

Promotion Logic​

  1. When a term in the cold layer is accessed:

    • Its access count is incremented
    • If access count ≥ medium_promotion_threshold (default: 5):
      • The term is moved to the medium layer
    • If access count ≥ hot_promotion_threshold (default: 100):
      • The term is moved directly to the hot layer

  2. When a term in the medium layer is accessed:

    • Its access count is incremented
    • If access count ≥ hot_promotion_threshold (default: 100):
      • The term is moved to the hot layer

Demotion Logic​

During periodic optimization:

  1. Terms in the hot layer with access count < hot_demotion_threshold (default: 10) are moved to the medium layer
  2. Terms in the medium layer with access count < medium_demotion_threshold (default: 1) are moved to the cold layer

Optimization Process​

The dictionary periodically optimizes itself (configurable, default every hour) by:

  1. Collecting all terms with their metadata
  2. Sorting by access frequency
  3. Redistributing terms to appropriate tiers
  4. Resetting the dictionary structure

Thread Safety​

The implementation uses RwLock to ensure thread safety:

  • Each layer is protected by its own lock
  • Read operations use shared locks
  • Write operations use exclusive locks
  • Fine-grained locking to minimize contention
  • Careful lock ordering to prevent deadlocks

Memory Efficiency​

Several techniques are employed to minimize memory usage:

String Interning​

  • Each unique string is stored exactly once
  • High-frequency terms use interned strings for comparison
  • Reduces memory for duplicate strings common in search applications

Tiered Storage​

  • Only the most accessed terms use higher memory data structures
  • Rare terms use more compact representations
  • Adaptive storage based on actual usage patterns

Serialization​

  • Custom serialization for compact persistence
  • Efficient deserialization for fast loading

Performance Characteristics​

Lookup Performance​

The tiered approach improves average lookup performance:

  • 80-90% of lookups hit the hot layer with O(1) performance
  • Medium-frequency terms use trie lookups with O(k) performance
  • Cold terms have standard HashMap lookup with O(1) average case

Memory Usage​

Memory efficiency improves as the dictionary learns the access patterns:

  • Frequently accessed terms use more memory per term but account for fewer terms
  • Infrequently accessed terms use less memory but account for more terms
  • String interning provides additional memory savings
  • Adaptive promotion ensures optimal memory allocation

Prefix Operations​

Prefix-based operations (needed for suggestions) are efficiently handled:

  • Prefix queries are primarily directed to the medium (trie) layer
  • Results are merged with any matching terms from other layers
  • Efficient implementation for autocomplete functionality

Configuration Parameters​

The Hybrid Dictionary can be customized through HybridDictionaryConfig (Rust) or DictionaryConfig (Python):

ParameterDescriptionDefault
hot_terms_capacityMaximum number of terms in hot layer10,000
hot_promotion_thresholdAccess count to promote to hot layer100
hot_demotion_thresholdAccess count below which hot terms are demoted10
medium_promotion_thresholdAccess count to promote from cold to medium5
medium_demotion_thresholdAccess count below which medium terms are demoted1
optimization_interval_secsSeconds between optimization runs3600

Configuring in Python​

You can configure the Hybrid Dictionary in Python using the new DictionaryConfig class:

from fast_inverted_index import Index, DictionaryConfig

# Create a custom dictionary configuration
dict_config = DictionaryConfig(
    hot_terms_capacity=5000,         # Adjust hot terms layer size
    hot_promotion_threshold=50,      # Lower threshold for promotion to hot layer
    medium_promotion_threshold=3,    # Lower threshold for promotion to medium layer
    optimization_interval_secs=1800  # Run optimization more frequently
)

# Apply configuration using builder pattern
builder = Index.builder()
builder.with_in_memory(True)
builder.with_dictionary_config(dict_config)
index = builder.build()

This allows you to fine-tune the dictionary's behavior for your specific workload.

Integration with Index​

The Hybrid Dictionary is integrated with the main Index class:

  • The IndexConfig class provides dictionary configuration options
  • The IndexBuilder has a new with_dictionary_config method
  • The IndexStats includes dictionary-specific metrics
  • All dictionary operations are transparently handled

The Hybrid Dictionary implements the same interface as the previous RadixTrie:

Implementation Details​

String Interning​

struct InternedString {
    content: Arc<String>,
}

impl InternedString {
    fn new(content: String) -> Self {
        Self {
            content: Arc::new(content),
        }
    }
    
    fn as_str(&self) -> &str {
        &self.content
    }
}

struct StringInterner {
    strings: HashMap<String, Arc<String>>,
}

The interning process:

  1. Check if string exists in the interner
  2. If present, return a shared reference to the existing string
  3. If not, create a new string and store it in the interner

Term Entry​

struct TermEntry<V> 
where
    V: Clone + Default + Serialize + for<'de> Deserialize<'de>,
{
    value: V,
    access_count: u64,
    last_access: u64,
}

Each term entry tracks:

  • The associated value
  • Number of accesses (used for promotion/demotion)
  • Last access timestamp (for potential time-based expiration)

Adaptive Promotion​

The core adaptive logic occurs in the get/insert methods:

// Simplified pseudocode for the promotion logic
if access_count >= hot_threshold {
    move_to_hot_layer(term);
} else if access_count >= medium_threshold {
    move_to_medium_layer(term);
} else {
    keep_in_cold_layer(term);
}

Concurrency Model​

The implementation uses a fine-grained concurrency approach:

  1. Each layer (hot, medium, cold) has its own RwLock
  2. The string interner has a separate RwLock
  3. Statistics are protected by their own RwLock
  4. Read operations acquire read locks
  5. Write operations acquire write locks
  6. Locks are acquired in a consistent order to prevent deadlocks
  7. Lock scope is minimized to reduce contention

Future Enhancements​

Potential improvements for future versions:

  1. Roaring Bitmaps: Integrate Roaring Bitmaps for frequent term posting lists
  2. SIMD Acceleration: Use SIMD instructions for term matching and posting list operations
  3. Segment-Based Storage: Implement immutable segments for improved write performance
  4. Fuzzy Matching: Advanced approximate matching for suggestions
  5. TTL-Based Eviction: Time-based term eviction for dynamic vocabulary environments

Conclusion​

The Hybrid Dictionary represents a significant advancement in the Fast-Inverted-Index library, addressing key performance bottlenecks while maintaining backward compatibility. Its adaptive, tiered approach provides an optimal balance of performance and memory efficiency, particularly for large-scale search applications with skewed term distributions.

The implementation demonstrates several advanced techniques:

  • Tiered storage based on access frequency
  • Dynamic adaptation to actual usage patterns
  • Memory optimization through string interning
  • Thread-safe concurrency model
  • Efficient serialization

These improvements provide a solid foundation for future enhancements and ensure the library can scale to meet the demands of production search applications.
