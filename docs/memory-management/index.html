Memory Management in Fast Inverted Index

This document covers best practices and APIs for managing memory usage in the fast-inverted-index library.

Overview​

The fast-inverted-index library maintains several in-memory data structures to achieve high performance:

  1. Posting Lists Cache: Stores recently accessed posting lists to improve search performance
  2. Field-Specific Memory Structures: Stores field-specific posting lists for indexed fields
  3. Document Metadata: Stores document information for quick access
  4. Dictionary: Maintains term-to-ID mappings with a tiered hot/medium/cold storage approach

While these structures are essential for performance, they can lead to increased memory usage in long-running applications. Version 0.4.4 introduces new tools to help manage memory effectively.

Memory Growth Patterns​

Memory usage can grow in the following ways:

  1. Document Count: As more documents are added, the index naturally requires more memory
  2. Cached Posting Lists: Recently accessed terms stay in cache even if not used frequently
  3. Field Proliferation: Indexes with many fields can accumulate memory in rarely used fields
  4. Complex Queries: Temporary objects created during complex query execution may persist

Memory Management APIs​

Rust API​

// In the Index struct
impl Index {
    // Manage memory usage by pruning cache and cleaning up rarely used fields
    pub fn manage_memory(&mut self) -> Result<()>;
    
    // Get a memory usage estimate for monitoring
    pub fn estimate_memory_usage(&self) -> HashMap<String, usize>;
    
    // Clean up resources after query execution
    pub fn cleanup_after_query(&self) -> Result<()>;
}

// In the LruCache implementation
impl<K, V> LruCache<K, V> {
    // Actively prune expired entries
    pub fn prune(&self) -> usize;
    
    // Get an estimate of memory usage
    pub fn estimate_size(&self) -> usize;
}

Python API​

# Index methods
class Index:
    # Manage memory usage by pruning cache and cleaning up rarely used fields
    def manage_memory(self): ...
    
    # Get memory usage statistics
    def memory_usage(self) -> dict: ...
    
    # Close the index and release resources
    def close(self): ...

Best Practices​

1. Proactive Memory Management​

For long-running applications, periodically call manage_memory() to prevent unbounded growth:

# Python example
import time

index = Index(storage_path="my_index")

# Add documents, perform searches...

# Every hour, clean up memory
while True:
    time.sleep(3600)  # 1 hour
    index.manage_memory()
// Rust example
use std::thread;
use std::time::Duration;

let mut index = Index::new()?;

// Add documents, perform searches...

// Clean up memory after batch operations
index.manage_memory()?;

2. Memory Monitoring​

Track memory usage to detect issues early:

# Python example
def log_memory_usage(index):
    memory = index.memory_usage()
    total_mb = memory["total"] / (1024 * 1024)
    print(f"Memory usage: {total_mb:.2f} MB")
    
    if total_mb > 1000:  # 1 GB
        print("Memory usage high, running cleanup")
        index.manage_memory()

3. Limit Batch Sizes​

Process documents in reasonable batches to allow for memory cleanup between batches:

# Python example
docs = load_documents()  # Large dataset

# Process in batches of 1000
for i in range(0, len(docs), 1000):
    batch = docs[i:i+1000]
    for doc in batch:
        index.add_document(doc["id"], doc["content"])
    
    # Manage memory after each batch
    index.manage_memory()

4. Proper Index Cleanup​

Always call close() when done with an index to release resources:

# Python example
try:
    index = Index(storage_path="my_index")
    # Use the index...
finally:
    # Ensure resources are released
    index.close()

5. Clean Up After Complex Queries​

For applications with complex, resource-intensive queries, clean up after each operation:

# Python example
# Execute a complex query
complex_query = QueryNode.AND([
    QueryNode.term("title", "rust"),
    QueryNode.OR([
        QueryNode.term("content", "performance"),
        QueryNode.term("content", "memory")
    ])
])

result = index.execute_query(complex_query)

# Process results...

# Clean up memory
index.manage_memory()

Memory Footprint Reduction Techniques​

  1. Use Field Selection: Only index the fields you need to search
  2. Be Selective with Storage: Use in_memory=False for persistent storage with RocksDB
  3. Set Cache Size Appropriately: Configure cache_size based on available memory
  4. Use TTL for Cache: Set cache_ttl_secs for automatic expiration of cached entries
  5. Specialized Indices: Create separate indices for different document types rather than one large index

Troubleshooting​

High Memory Usage​

If memory usage is higher than expected:

  1. Call memory_usage() to identify which components are using memory
  2. Call manage_memory() to clean up unused resources
  3. Reduce the cache_size parameter when creating the index
  4. Set a shorter cache TTL with cache_ttl_secs
  5. Check if you're storing unnecessary fields

Cannot Close Index​

If you cannot close the index due to reference errors:

  1. Make sure all query results and other objects are no longer referenced
  2. Force garbage collection by calling gc.collect() in Python
  3. Check for circular references or leaked objects

Implementation Details​

The memory management features use the following techniques:

  1. The LRU cache uses Moka's run_pending_tasks() to prune expired entries
  2. Field memory management identifies rarely used fields based on document counts
  3. Memory usage estimation uses approximations based on collection sizes
  4. The close method attempts to get exclusive ownership of the index to clean up

By following these guidelines, you can use the fast-inverted-index library in long-running applications without memory concerns.
