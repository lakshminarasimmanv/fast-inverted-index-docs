Observability & Monitoring

Fast-Inverted-Index provides comprehensive observability capabilities to help you monitor and debug your search system in production environments.

Overview​

The observability system consists of three main components:

  1. Metrics: Production-grade metrics using Prometheus for collection and export
  2. Health Checks: Health status reporting for system components
  3. Logging: Structured logging with contextual information

Getting Started​

To start using the observability features, you need to create an ObservabilityContext and connect it to your index:

use fast_inverted_index::{Index, IndexBuilder, ObservabilityContext};

// Create observability context
let observability = ObservabilityContext::new();

// Create your index
let mut index = Index::new().unwrap();

// Connect the index metrics with observability
let obs_metrics = observability.metrics.clone();
index.metrics.with_observability(obs_metrics);

Metrics System​

Fast-Inverted-Index provides a comprehensive metrics system based on Prometheus. The metrics are designed to help you understand the performance and behavior of your index in production.

Exposing Metrics​

To expose Prometheus metrics via HTTP in Rust:

use prometheus_http_api::HttpServer;

// Start a Prometheus HTTP server
let server = HttpServer::new(observability.registry());
server.start("0.0.0.0:9091").unwrap();

For Python applications:

from prometheus_client import start_http_server, Gauge, Counter

# Set up Prometheus metrics
DOCUMENTS_TOTAL = Gauge('fii_documents_total', 'Total number of documents indexed')
TERMS_TOTAL = Gauge('fii_terms_total', 'Total number of terms in the index')

def update_prometheus_metrics(index):
    """Update Prometheus metrics from index stats"""
    stats = index.stats()
    DOCUMENTS_TOTAL.set(stats['document_count'])
    TERMS_TOTAL.set(stats['term_count'])

# Start Prometheus server
start_http_server(9090)

Integration with Prometheus​

Add your metrics endpoint to your Prometheus configuration:

scrape_configs:
  - job_name: 'fast-inverted-index'
    scrape_interval: 15s
    static_configs:
      - targets: ['your-server:9091']

Health Checks​

The health check system provides status information about different components of your index:

use fast_inverted_index::health::{HealthStatus, ComponentHealth};

// Check overall health
let status = observability.check_health();
if status != HealthStatus::Healthy {
    println!("System health is degraded!");
}

// Add or update component health
observability.health.update_component(
    "cache",
    ComponentHealth::healthy().with_metric("hit_rate", 0.95)
);

// Get a detailed health report
let report = observability.health.health_report();
println!("{}", report);  // JSON formatted health report

Health Status Formats​

The health report is returned in JSON format, making it easy to integrate with monitoring systems:

{
  "status": "Healthy",
  "last_checked": "1609459200",
  "components": {
    "index": {
      "status": "Healthy",
      "metrics": {
        "metric_document_count": "1000000",
        "metric_avg_query_time_ms": "1.23"
      }
    },
    "cache": {
      "status": "Healthy",
      "metrics": {
        "metric_hit_rate": "0.95"
      }
    }
  }
}

Accessing Metrics in Python​

In Python applications, you can access metrics through the stats() method:

from fast_inverted_index import Index

# Create an index
index = Index()

# Get metrics
stats = index.stats()
print(f"Documents indexed: {stats['document_count']}")
print(f"Average query time: {stats['avg_query_time_ms']} ms")
print(f"Cache hit rate: {stats['cache_hit_rate']}")

Recommended Dashboards​

For monitoring in production, we recommend creating Grafana dashboards with the following panels:

  1. Query Performance:

    • Query latency (p50, p95, p99)
    • Queries per second
    • Query types distribution

  2. Index Operations:

    • Documents indexed per second
    • Indexing latency
    • Index size growth

  3. Cache Performance:

    • Cache hit rate
    • Cache size
    • Cache evictions

  4. Dictionary Performance:

    • Terms by tier (hot/medium/cold)
    • Dictionary hit rate

  5. Health Status:

    • Component health status
    • Failed health checks

Example Use Cases​

Monitoring Query Performance​

// Every query is automatically timed and recorded
let timer = index.metrics.start_typed_query_timer("term");
let results = index.term_query("example");
// Timer automatically records when it goes out of scope

Tracking Index Growth​

The metrics automatically track document and term counts:

// Adding documents automatically updates metrics
index.add_document(1, "Example document").unwrap();

// You can view stats anytime
println!("Indexed {} documents", index.stats().document_count);

Health Checks for Load Balancers​

// Create a simple health check endpoint
fn health_endpoint() -> Response {
    let status = observability.check_health();
    
    match status {
        HealthStatus::Healthy => Response::ok("Healthy"),
        HealthStatus::Degraded => Response::status(200, "Degraded"),
        HealthStatus::Unhealthy => Response::status(503, "Unhealthy"),
    }
}

Best Practices​

  1. Monitor Cache Hit Rates: Keep track of your cache hit rates. If they fall below 80%, consider increasing cache size.

  2. Watch Dictionary Tier Distribution: The dictionary's performance depends on having hot terms in the hot tier. Monitor this distribution.

  3. Set Up Alerts: Configure alerts for unusual patterns:

    • Query latency spikes
    • High error rates
    • Disk/memory usage above thresholds
    • Cache hit rate below threshold

  4. Periodic Optimization: Schedule index optimizations during low-traffic periods.

  5. Resource Monitoring: Always monitor memory and disk usage alongside index-specific metrics.

Validation​

The repository includes validation scripts to verify the observability system is working correctly:

  1. test_observability.py: Basic tests to validate metrics are being tracked
  2. validate_observability.py: Comprehensive validation of metrics behavior under load

Run the validation script before deploying to production:

python validate_observability.py

This will ensure the observability features are reporting accurately for your deployment.
