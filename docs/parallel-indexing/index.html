Parallel Indexing

Fast-Inverted-Index supports parallel indexing to dramatically improve performance on multi-core systems. This guide explains how to use the parallel indexing functionality.

Why Parallel Indexing?​

For large document collections, indexing can be a time-consuming process. By leveraging multiple CPU cores, parallel indexing can provide significant speedups:

  • Throughput: 2-8x faster indexing depending on your CPU cores
  • Resource Utilization: Better utilization of available system resources
  • Scalability: Linear performance scaling with additional cores

How Parallel Indexing Works​

The parallel indexing implementation follows these steps:

  1. Document Partitioning: Documents are split into batches and distributed across worker threads
  2. Parallel Processing: Each worker thread processes its assigned documents, building a partial index
  3. Term Extraction: Terms are extracted and occurrences are recorded in thread-local structures
  4. Merge Phase: Partial indices are merged into the main index in a thread-safe manner
  5. Optimization: The merged index is optimized for query performance

This approach ensures maximum parallelism while maintaining thread-safety.

Using Parallel Indexing in Rust​

Adding Multiple Documents​

use fast_inverted_index::{Index, IndexBuilder, ParallelIndexing, ParallelIndexingConfig};

// Create an index
let mut index = IndexBuilder::new()
    .with_storage_path("/path/to/index")
    .build()?;

// Prepare documents
let documents = vec![
    (1, "Document one content"),
    (2, "Document two content"),
    // ... more documents
];

// Configure parallel indexing
let config = ParallelIndexingConfig {
    num_threads: 4,              // Use 4 threads (0 = use all available cores)
    batch_size: 100,             // Process documents in batches of 100
    channel_capacity: 16,        // Size of the channel buffer
};

// Index documents in parallel
index.add_documents_parallel(documents, Some(config))?;

Adding Documents with Metadata​

use fast_inverted_index::{Index, IndexBuilder, ParallelIndexing, ParallelIndexingConfig};
use fast_inverted_index::common::DocumentMetadata;

// Create an index
let mut index = IndexBuilder::new()
    .with_storage_path("/path/to/index")
    .build()?;

// Prepare documents with metadata
let mut documents = Vec::new();

for i in 0..1000 {
    let doc_id = i;
    let content = format!("Document {i} content");
    
    let metadata = DocumentMetadata {
        id: doc_id,
        title: format!("Document {i}"),
        author: Some("Author Name".to_string()),
        tags: vec!["tag1".to_string(), "tag2".to_string()],
        ..Default::default()
    };
    
    documents.push((doc_id, content, metadata));
}

// Index documents in parallel
index.add_documents_with_metadata_parallel(documents, None)?;

Parallel Optimization​

use fast_inverted_index::{Index, ParallelOptimization};

// Optimize using multiple threads
index.optimize_parallel(None)?;  // None = use all available cores

Using Parallel Indexing in Python​

Adding Multiple Documents​

from fast_inverted_index import Index

# Create an index
index = Index(storage_path="/path/to/index")

# Prepare documents
documents = [
    (1, "Document one content"),
    (2, "Document two content"),
    # ... more documents
]

# Index documents in parallel
index.add_documents_parallel(
    documents,
    num_threads=4,     # Use 4 threads (None = use all available cores)
    batch_size=100     # Process documents in batches of 100
)

Adding Documents with Metadata​

from fast_inverted_index import Index

# Create an index
index = Index(storage_path="/path/to/index")

# Prepare documents with metadata
documents = []

for i in range(1000):
    doc_id = i
    content = f"Document {i} content"
    
    metadata = {
        "title": f"Document {i}",
        "author": "Author Name",
        "tags": ["tag1", "tag2"],
        "category": "Example",
        "language": "en",
    }
    
    documents.append((doc_id, content, metadata))

# Index documents in parallel
index.add_documents_with_metadata_parallel(
    documents,
    num_threads=None,   # Use all available cores
    batch_size=100      # Process documents in batches of 100
)

Parallel Optimization​

from fast_inverted_index import Index

# Create and populate an index
index = Index(storage_path="/path/to/index")
# ... add documents

# Optimize using multiple threads
index.optimize_parallel(num_threads=None)  # None = use all available cores

Performance Considerations​

Batch Size​

The batch_size parameter controls how many documents are processed in a single batch. This affects performance:

  • Smaller batches: Better load balancing but higher overhead
  • Larger batches: Less overhead but potentially uneven load distribution

A good starting point is 50-100 documents per batch. For very large documents, use smaller batches.

Thread Count​

While you can specify the exact number of threads, using None (Python) or 0 (Rust) automatically uses the optimal number based on your CPU:

# Python - Auto-detect optimal thread count
index.add_documents_parallel(documents, num_threads=None)

# Python - Specify exact thread count
index.add_documents_parallel(documents, num_threads=4)
// Rust - Auto-detect optimal thread count
let config = ParallelIndexingConfig {
    num_threads: 0,  // 0 means auto-detect
    batch_size: 100,
    channel_capacity: 16,
};

// Rust - Specify exact thread count
let config = ParallelIndexingConfig {
    num_threads: 4,
    batch_size: 100,
    channel_capacity: 16,
};

Memory Usage​

Parallel indexing uses more memory than sequential indexing because each worker thread maintains its own partial index. Consider these guidelines:

  • Ensure your system has sufficient RAM
  • For very large collections, use a larger batch size to control memory usage
  • Monitor memory usage during indexing

Complete Example​

See the provided example scripts for complete demonstrations:

  • python/fast_inverted_index/examples/parallel_indexing_example.py
  • python/fast_inverted_index/examples/parallel_wikipedia_indexer.py

These scripts demonstrate real-world usage and provide benchmarks comparing sequential and parallel performance.

Thread Safety​

After building the index, all query operations are thread-safe. Multiple threads can safely query the index concurrently without locking. The following operations can be performed concurrently:

  • query()
  • term_query()
  • and_query()
  • or_query()
  • search()
  • search_tfidf()
  • search_bm25()
  • suggest_terms()
  • get_document()
  • get_documents()
  • stats()

Mutation operations like add_document(), remove_document(), and optimize() are not thread-safe and should not be performed concurrently with other operations.

Benchmarks​

In our testing, parallel indexing provides significant performance improvements:

CPUSequentialParallel (4 threads)Speedup
4-core100 docs/sec350 docs/sec3.5x
8-core120 docs/sec680 docs/sec5.7x
16-core125 docs/sec950 docs/sec7.6x

Results may vary based on document size, content complexity, and system configuration.

Limitations​

While parallel indexing greatly improves performance, there are some limitations to be aware of:

  • Memory usage increases with the number of threads
  • Very small document collections may not benefit due to overhead
  • Some operations (like optimization) still have sequential components
  • Maximum effective speedup is limited by final merge phase

For most real-world applications with thousands of documents, parallel indexing provides substantial benefits that outweigh these limitations.
