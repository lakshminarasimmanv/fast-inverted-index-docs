Performance Tuning

Fast-Inverted-Index is designed for high performance out of the box, but you can further optimize it for specific use cases. This guide covers various performance tuning strategies.

Configuration Parameters​

Dictionary Configuration​

The hybrid dictionary can be configured for optimal performance:

# Python - Configure the hybrid dictionary
index = Index.builder() \
    .with_dictionary_config({
        "hot_terms_capacity": 20000,           # Size of hot terms layer
        "hot_promotion_threshold": 100,         # Access count to promote to hot layer
        "hot_demotion_threshold": 10,           # Access count below which hot terms are demoted
        "medium_promotion_threshold": 5,         # Access count to promote from cold to medium
        "medium_demotion_threshold": 1,          # Access count below which medium terms are demoted
        "optimization_interval_secs": 3600       # How often to optimize the dictionary
    }) \
    .build()
// Rust - Configure the hybrid dictionary
let index = IndexBuilder::new()
    .with_dictionary_config(HybridDictionaryConfig {
        hot_terms_capacity: 20000,
        hot_promotion_threshold: 100,
        hot_demotion_threshold: 10,
        medium_promotion_threshold: 5,
        medium_demotion_threshold: 1,
        optimization_interval_secs: 3600,
    })
    .build()?;

Guidelines:

  • For small indices (< 100K documents): Hot terms capacity of 1,000-5,000
  • For medium indices (100K-1M documents): Hot terms capacity of 5,000-20,000
  • For large indices (> 1M documents): Hot terms capacity of 20,000-100,000
  • Adjust promotion/demotion thresholds based on query patterns

Cache Size​

The LRU cache size impacts query performance significantly:

# Python - Large cache for better performance
index = Index(cache_size=500000)
// Rust - Large cache for better performance
let index = IndexBuilder::new()
    .with_cache_size(500000)
    .build()?;

Guidelines:

  • For small indices (< 100K documents): 10,000-50,000 cache entries
  • For medium indices (100K-1M documents): 50,000-500,000 cache entries
  • For large indices (> 1M documents): 500,000+ cache entries
  • Monitor memory usage and adjust accordingly

Cache TTL​

Set an appropriate time-to-live for cached entries:

# Python - 1 hour TTL
index = Index(cache_ttl_secs=3600)
// Rust - 1 hour TTL
use std::time::Duration;
let index = IndexBuilder::new()
    .with_cache_ttl(Duration::from_secs(3600))
    .build()?;

Guidelines:

  • For mostly static data: Longer TTL (hours)
  • For frequently changing data: Shorter TTL (minutes)
  • For real-time applications: Very short TTL (seconds)

In-Memory Mode​

For maximum performance at the cost of persistence:

# Python - In-memory mode
index = Index(in_memory=True)
// Rust - In-memory mode
let index = IndexBuilder::new()
    .with_in_memory(true)
    .build()?;

This mode is beneficial for:

  • Testing and development
  • Temporary indices
  • Short-lived applications
  • Scenarios where persistence is handled externally

Position Storage​

Disable position storage if you don't need phrase queries:

# Python - Disable position storage
index = Index(store_positions=False)
// Rust - Disable position storage
let index = IndexBuilder::new()
    .with_store_positions(false)
    .build()?;

This can reduce index size and improve indexing speed significantly.

Bulk Operations​

Bulk Indexing​

For large document sets, use batch processing:

# Python - Bulk indexing
start_time = time.time()

for i, doc in enumerate(documents):
    index.add_document(i, doc["content"])
    
    # Periodically optimize
    if i > 0 and i % 10000 == 0:
        index.optimize()
        print(f"Indexed {i} documents...")

elapsed_time = time.time() - start_time
print(f"Indexed {len(documents)} documents in {elapsed_time:.2f} seconds "
      f"({len(documents) / elapsed_time:.2f} docs/sec)")
// Rust - Bulk indexing
let start = std::time::Instant::now();
let mut count = 0;

for (i, doc) in documents.iter().enumerate() {
    index.add_document(i as u64, &doc.content)?;
    count += 1;
    
    // Periodically optimize
    if i > 0 && i % 10000 == 0 {
        index.optimize()?;
        println!("Indexed {} documents...", i);
    }
}

let elapsed = start.elapsed();
println!("Indexed {} documents in {:.2?} ({:.2} docs/sec)",
         count, elapsed, count as f64 / elapsed.as_secs_f64());

Parallel Indexing​

For multi-core systems, distribute indexing across processes:

# Python - Parallel indexing
from concurrent.futures import ProcessPoolExecutor
import os

def index_document_batch(batch):
    # Create a separate index for each process
    idx = Index(in_memory=True)
    for doc in batch:
        idx.add_document(doc["id"], doc["content"])
    return idx.stats()

# Split documents into batches
def chunks(lst, n):
    for i in range(0, len(lst), n):
        yield lst[i:i + n]

# Number of processes
num_processes = os.cpu_count()
print(f"Using {num_processes} processes")

# Batch size
batch_size = max(1, len(documents) // num_processes)
batches = list(chunks(documents, batch_size))

# Process in parallel
with ProcessPoolExecutor(max_workers=num_processes) as executor:
    futures = [executor.submit(index_document_batch, batch) for batch in batches]
    results = [future.result() for future in futures]

# Show results
total_docs = sum(r["document_count"] for r in results)
total_terms = sum(r["term_count"] for r in results)
print(f"Indexed {total_docs} documents with {total_terms} terms")

Query Optimization​

Query Construction​

Choose the right query type:

# Python - Single term (fastest)
results = index.term_query("word")

# AND query (faster than OR)
results = index.and_query(["word1", "word2"])

# OR query (slower)
results = index.or_query(["word1", "word2"])
// Rust - Single term (fastest)
let query = Query::term("word");

// AND query (faster than OR)
let query = Query::and(vec!["word1".to_string(), "word2".to_string()]);

// OR query (slower)
let query = Query::or(vec!["word1".to_string(), "word2".to_string()]);

Field Boosting​

Optimize relevance with field boosting:

# Python - Boost important fields
results = index.search_with_metadata(
    "query terms",
    ranking_method="bm25",
    boost_fields={
        "title": 3.0,     # Title is most important
        "tags": 2.0,      # Tags are very relevant
        "summary": 1.5,   # Summary is somewhat relevant
        # Content has default weight of 1.0
    }
)
// Rust - Boost important fields
let mut field_boosts = std::collections::HashMap::new();
field_boosts.insert("title".to_string(), 3.0);
field_boosts.insert("tags".to_string(), 2.0);
field_boosts.insert("summary".to_string(), 1.5);

let query = Query::parse("query terms");
let results = index.search_bm25(&query, Some(field_boosts))?;

Term Frequency Impact​

Be aware of term frequency impact:

  • Common terms (high frequency) get lower relevance scores
  • Rare terms (low frequency) get higher relevance scores
  • For exact matching, consider using term_query instead of ranked search

Storage Optimization​

RocksDB Settings​

For persistent storage, optimize RocksDB parameters:

// Rust - Custom RocksDB options
use rocksdb::{Options, DB};

let mut options = Options::default();
options.set_max_open_files(1000);
options.set_use_fsync(false);
options.set_bytes_per_sync(8388608); // 8MB
options.set_compression_type(rocksdb::DBCompressionType::Lz4);
options.set_compaction_style(rocksdb::DBCompactionStyle::Level);
options.set_write_buffer_size(64 * 1024 * 1024); // 64MB
options.set_max_write_buffer_number(4);
options.set_target_file_size_base(64 * 1024 * 1024); // 64MB

// Currently these advanced options need to be set through environment variables
// until a more flexible API is available
std::env::set_var("ROCKSDB_MAX_OPEN_FILES", "1000");
std::env::set_var("ROCKSDB_BYTES_PER_SYNC", "8388608");

Compression Settings​

Balance between storage size and performance:

  • Varint compression is used by default for efficient storage
  • For maximum performance, use the in-memory mode
  • For better compression, use the persistent storage with a smaller block size

Memory Management​

Document Size​

Keep document sizes reasonable:

  • Break very large documents into smaller chunks
  • If using positions, limit content to a reasonable size (e.g., < 100KB per document)
  • For very large text, consider extracting key sections or summarizing
# Python - Handle large documents by chunking
def index_large_document(index, doc_id, content, chunk_size=50000):
    """Index a large document by breaking it into chunks."""
    chunks = [content[i:i+chunk_size] for i in range(0, len(content), chunk_size)]
    
    # Index each chunk with the same document ID
    for i, chunk in enumerate(chunks):
        metadata = {
            "title": f"Document {doc_id} - Chunk {i+1}",
            "parent_id": doc_id,
            "chunk_index": i,
            "total_chunks": len(chunks)
        }
        index.add_document(f"{doc_id}_{i}", chunk, metadata)

Metadata Size​

Keep metadata fields concise:

  • Large arrays (e.g., thousands of tags) can impact performance
  • Keep string fields to a reasonable size
  • Use references to external storage for large binary data

Monitor index statistics to identify performance issues:

stats = index.stats()
print(f"Documents: {stats['document_count']}")
print(f"Terms: {stats['term_count']}")
print(f"Cache hit rate: {stats['cache_hit_rate']}")

# Dictionary statistics
if 'dictionary' in stats:
    dict_stats = stats['dictionary']
    print("\nDictionary Statistics:")
    print(f"Total terms: {dict_stats['total_terms']}")
    print(f"Hot terms: {dict_stats['tier_distribution']['hot']}")
    print(f"Medium terms: {dict_stats['tier_distribution']['medium']}")
    print(f"Cold terms: {dict_stats['tier_distribution']['cold']}")
    print(f"Dictionary hit rate: {dict_stats['hit_rate']:.2f}")
// Rust - Monitor statistics
let stats = index.stats()?;
println!("Documents: {}", stats.document_count);
println!("Terms: {}", stats.term_count);
println!("Cache hit rate: {}", stats.cache_hit_rate);
println!("Avg query time: {} ms", stats.avg_query_time_ms);
println!("Avg indexing time: {} ms", stats.avg_indexing_time_ms);

// Dictionary statistics
if let Some(dict_stats) = stats.dictionary_stats {

import time

doc_count = 10000

Scaling Strategies​

Vertical Scaling​

Optimize for a single machine:

  • Increase cache size for better performance
  • Use in-memory mode if data fits in RAM
  • Optimize RocksDB settings for your storage
  • Periodically run optimize() to compact the index

Horizontal Scaling​

For very large datasets, consider these strategies:

  1. Sharding by Document ID:

    • Split documents across multiple indices based on ID ranges
    • Query each shard and merge results

  2. Sharding by Term:

    • Split terms across multiple indices
    • Route queries to appropriate shards based on terms

  3. Replication:

    • Maintain identical copies of the index on multiple machines
    • Use for read scaling and fault tolerance

Example of a simple document-based sharding strategy:

class ShardedIndex:
    def __init__(self, shard_count=4, storage_base_path=None):
        """Initialize a sharded index."""
        self.shard_count = shard_count
        self.shards = []
        
        for i in range(shard_count):
            storage_path = None
            if storage_base_path:
                storage_path = f"{storage_base_path}/shard_{i}"
            
            self.shards.append(Index(storage_path=storage_path))
    
    def add_document(self, doc_id, content, metadata=None):
        """Add a document to the appropriate shard."""
        shard_index = doc_id % self.shard_count
        self.shards[shard_index].add_document(doc_id, content, metadata)
    
    def search(self, query):
        """Search across all shards and merge results."""
        all_results = []
        
        for shard in self.shards:
            results = shard.search(query, ranking_method="bm25")
            all_results.extend(results)
        
        # Sort by score (higher is better)
        all_results.sort(key=lambda x: x[1], reverse=True)
        return all_results

Advanced Optimization​

Custom Tokenization​

For specialized domains, implement custom tokenization:

# Python - Custom tokenization example
def custom_tokenize(text):
    """Custom tokenization for specialized domains."""
    # Remove common noise
    text = re.sub(r'[^\w\s]', ' ', text.lower())
    
    # Handle domain-specific terms
    text = re.sub(r'c\+\+', 'cpp', text)
    text = re.sub(r'\.net', 'dotnet', text)
    
    # Split and filter
    tokens = text.split()
    tokens = [t for t in tokens if len(t) > 1]
    
    return tokens

# Use with a pre-processor
for doc in documents:
    processed_content = ' '.join(custom_tokenize(doc['content']))
    index.add_document(doc['id'], processed_content)

Result Caching​

For frequently accessed queries, implement a result cache:

class CachedSearchEngine:
    def __init__(self, index, cache_size=1000):
        """Initialize a search engine with query result caching."""
        self.index = index
        self.cache = {}
        self.cache_size = cache_size
        self.cache_hits = 0
        self.cache_misses = 0
    
    def search(self, query, force_refresh=False):
        """Search with caching."""
        # Normalize the query for cache key
        cache_key = query.lower().strip()
        
        # Check cache unless refresh is forced
        if not force_refresh and cache_key in self.cache:
            self.cache_hits += 1
            return self.cache[cache_key]
        
        # Execute the query
        self.cache_misses += 1
        results = self.index.search(query, ranking_method="bm25")
        
        # Cache the results
        self.cache[cache_key] = results
        
        # Maintain cache size
        if len(self.cache) > self.cache_size:
            # Simple LRU: remove the first item (oldest)
            self.cache.pop(next(iter(self.cache)))
        
        return results
    
    def cache_stats(self):
        """Get cache statistics."""
        total = self.cache_hits + self.cache_misses
        hit_rate = self.cache_hits / total if total > 0 else 0
        return {
            "size": len(self.cache),
            "capacity": self.cache_size,
            "hits": self.cache_hits,
            "misses": self.cache_misses,
            "hit_rate": hit_rate
        }

Parallel Processing​

For significantly improved indexing performance on multi-core systems, use the parallel indexing capabilities:

# Python - Parallel document indexing
index.add_documents_parallel(documents, num_threads=None, batch_size=100)

# Python - Parallel optimization
index.optimize_parallel()
// Rust - Parallel document indexing
let config = ParallelIndexingConfig {
    num_threads: 0,  // 0 = auto-detect
    batch_size: 100,
    channel_capacity: 16,
};

index.add_documents_parallel(documents, Some(config))?;

// Rust - Parallel optimization
index.optimize_parallel(None)?;

For detailed information on parallel processing, see the Parallel Indexing guide.

Production Readiness Checklist​

  • Configure appropriate cache size and TTL
  • Enable parallel indexing for multi-core systems
  • Set up periodic index optimization
  • Implement monitoring for index statistics
  • Test query performance with realistic data volumes
  • Implement appropriate error handling
  • Set up backup and recovery procedures
  • Consider sharding for very large datasets
  • Tune RocksDB settings for persistent storage
  • Implement result caching for common queries
  • Monitor memory usage in production

By following these performance tuning guidelines, you can ensure that your Fast-Inverted-Index implementation delivers optimal performance for your specific use case.
