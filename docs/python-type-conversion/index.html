Python Type Conversion in Fast Inverted Index

This document explains how types are converted between Python and Rust in the Fast Inverted Index library, focusing on fields like tags, document metadata, and search parameters.

Version 0.4.6 Type Handling Improvements​

Fast Inverted Index 0.4.6 includes comprehensive improvements to the type conversion system to handle common Python data types more robustly:

  1. Tag field handling (supporting both string and list formats)
  2. Document metadata preservation during parallel indexing
  3. Batch document processing with mixed field types
  4. Enhanced search parameters and results
  5. Better error messages for conversion failures

Fields and Metadata Handling​

Tags Field​

The tags field is specially handled to support both string and list formats:

# As a String (Space-Separated)
index.add_document(1, "Document content", {
    "tags": "tag1 tag2 tag3"
})

# As a List of Strings
index.add_document(1, "Document content", {
    "tags": ["tag1", "tag2", "tag3"]
})

When retrieving documents, tags are always returned as a list of strings for consistency.

Supported Field Types​

The library supports the following field types:

Field TypePython TypeExample
TextString"Document content"
TagsString or List[String]"tag1 tag2" or ["tag1", "tag2"]
Numericint, float42, 3.14
BooleanboolTrue, False
Timestampintint(time.time())

Document Metadata Preservation​

A significant improvement in version 0.4.6 is the preservation of document metadata during parallel indexing:

  1. Existing metadata is preserved when merging partial indices
  2. Timestamps and other metadata fields are properly updated during indexing
  3. Field values are consistently handled across different operation types

Batch Document Processing​

The library now robustly handles batch document additions with mixed field types:

# Add multiple documents with different field formats
documents = [
    (1, {"content": "Document 1", "tags": ["tag1", "tag2"]}),
    (2, {"content": "Document 2", "tags": "tag3 tag4"})
]

# Format is (doc_id, fields_dict)
index.add_documents_with_fields_parallel(documents)

Each document's fields are properly converted and preserved during batch processing.

Query Building and Search​

QueryBuilder API​

The QueryBuilder provides a convenient way to build boolean queries:

# Create a QueryBuilder with an index
builder = QueryBuilder(index)

# Add terms to the builder (returns the builder for method chaining)
builder.add_term("term1").add_term("term2")

# Build an OR query (returns document IDs that match ANY of the terms)
doc_ids = builder.build_or_query()  # Returns a list of doc_ids

# Build an AND query (returns document IDs that match ALL of the terms)
doc_ids = builder.build_and_query()  # Returns a list of doc_ids

# Clear terms for reuse
builder.clear().add_term("new_term")

For complete documentation of the QueryBuilder API, see the dedicated Query Builder documentation.

Search Methods​

The library provides several ways to search:

# String-based search (returns list of (doc_id, score) tuples)
results = index.search("term1 term2")  # Default AND behavior
results = index.search("term1 OR term2")  # OR behavior

# With parameters
results = index.search("term1", 
                      limit=10,  # Limit results
                      boost_fields={"title": 3.0, "tags": 1.5})  # Field boosting

# QueryNode API for advanced queries
from fast_inverted_index import QueryNode, QueryExecutionParams

query = QueryNode.OR([
    QueryNode.term("title", "term1", boost=2.0),
    QueryNode.term("content", "term2")
])

params = QueryExecutionParams(scoring_algorithm="bm25l", limit=10)
result = index.execute_query(query, params)

Helper Methods​

The library provides helper methods to validate and prepare document metadata:

validate_document​

Checks if a document's metadata is valid before adding it:

is_valid, errors = index.validate_document(doc_id, content, metadata)
if not is_valid:
    print(f"Document validation errors: {errors}")

prepare_metadata​

Converts metadata to a format acceptable by the indexer:

# Convert fields to indexer-friendly format
prepared = index.prepare_metadata(metadata)

Type Conversion Workflow​

Follow this recommended workflow to avoid type conversion issues:

  1. Validate documents before indexing:

    is_valid, errors = index.validate_document(doc_id, content, metadata)
    if not is_valid:
        print(f"Document {doc_id} is invalid: {errors}")

  2. Convert metadata to ensure compatibility:

    compatible_metadata = index.prepare_metadata(metadata)
    index.add_document(doc_id, content, compatible_metadata)

  3. Use proper types for each field:

    • title, content, summary, category, language: String
    • tags: String (space-separated) or List of strings
    • created_at, updated_at: Integer timestamp

Error Handling​

The library provides descriptive error messages to help diagnose type conversion issues:

  • "Field 'tags' expected a string or list of strings"
  • "Field value cannot be a dictionary. Convert to a string representation first."
  • "Metadata must be a dictionary"
  • "Document already exists: {id}"

Best Practices​

For optimal type handling:

  1. For tags: Use either list format ["tag1", "tag2"] or space-separated strings "tag1 tag2"
  2. For batch operations: Be consistent with field types across documents
  3. For search: Use raw query strings for simple operations and QueryNode for complex queries
  4. For field boosting: Use the boost_fields parameter: index.search("query", boost_fields={"title": 2.0})
  5. For complex data types: Convert to strings before indexing (e.g., using json.dumps())
  6. For content storage: Use external storage like Redis for document content as shown in Document Content Storage

Thread Safety​

All type conversion operations are thread-safe, allowing for concurrent document addition and searching without type conversion issues. The thread safety improvements in version 0.4.5 and above ensure reliable operation in multi-threaded environments.
