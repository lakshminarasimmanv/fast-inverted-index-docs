Query Engine

The Query Engine is a powerful component that provides advanced query capabilities for the inverted index. It introduces a query DSL (Domain Specific Language) for building complex queries with flexible scoring options.

Overview​

The Query Engine provides:

  • A flexible query node tree structure for complex queries
  • Support for term, phrase, prefix, fuzzy, and range queries
  • Boolean operators (AND, OR, NOT) for combining queries
  • Field-specific searching with boosting
  • Multiple scoring algorithms (TF-IDF, BM25, BM25L, Learning-to-Rank)
  • Score explanation system for debugging and understanding relevance

Rust API​

QueryNode​

The QueryNode enum represents different types of queries that can be composed into a tree structure:

Important Note about Range Queries: When creating range queries, always use the direct field name (e.g., "word_count") and not the nested path (e.g., "metadata.word_count"), even though the field may appear nested in document structure. Using nested paths (like "metadata.field_name") with range queries will not return the expected results.

// Create a simple term query
let query = QueryNode::term("content", "search");

// Create a boosted term query
let query = QueryNode::term_with_boost("title", "search", 2.0);

// Create a phrase query with slop (words can be up to 2 positions apart)
let query = QueryNode::phrase("content", vec!["fast", "search"], 2);

// Create a prefix query (matches words starting with "prog")
let query = QueryNode::prefix("content", "prog");

// Create a fuzzy query (allowing 1 edit distance)
let query = QueryNode::fuzzy("content", "search", 1);

// Create a range query for numeric field (documents with 100-500 words)
// IMPORTANT: Use direct field name, not the nested path shown in document structure
let query = QueryNode::range(
    "word_count",  // Use direct field name, not "metadata.word_count"
    Bound::Included(Value::Integer(100)), 
    Bound::Excluded(Value::Integer(500))
);

// Combine with boolean operators
let query = QueryNode::and(vec![
    QueryNode::term("content", "search"),
    QueryNode::term("title", "engine"),
]);

// NOT query (documents without the term "slow")
let query = QueryNode::not(
    QueryNode::term("content", "slow")
);

// Filter query (search in content, filtered by category)
let query = QueryNode::filter(
    QueryNode::term("content", "search"),
    QueryNode::term("category", "technology")
);

QueryExecutor and QueryEngine​

The QueryExecutor trait defines methods for executing queries against the index, while the QueryEngine provides the implementation:

// Create a query engine for an index
let engine = QueryEngine::new(&index);

// Create execution parameters
let params = QueryExecutionParams {
    scoring_algorithm: ScoringAlgorithm::BM25,
    limit: Some(10),
    explain: false,
    ..Default::default()
};

// Execute a query
let result = engine.execute(&query, &params)?;

// Access the scored results
for (doc_id, score) in &result.scored_docs {
    println!("Document {}: score {}", doc_id, score);
}

// Get explanation for a document's score
let explanation = engine.explain(&query, doc_id, &params)?;
println!("{}", explanation.to_string_tree());

Combining with the Index API​

The Query Engine is designed to work seamlessly with the existing Index API:

// Execute a query with BM25 scoring
let result = index.execute_query(&query, None)?;

// Execute with custom parameters
let params = QueryExecutionParams {
    scoring_algorithm: ScoringAlgorithm::BM25L,
    limit: Some(20),
    field_boosts: {
        let mut boosts = HashMap::new();
        boosts.insert("title".to_string(), 2.0);
        boosts.insert("tags".to_string(), 1.5);
        boosts
    },
    ..Default::default()
};

let result = index.execute_query(&query, Some(params))?;

QueryBuilder​

A fluent builder API is provided for constructing queries:

let query = QueryBuilder::new()
    .term("content", "search")
    .boost(1.5)
    .build()
    .unwrap();

// Create a complex query
let title_query = QueryBuilder::new()
    .term("title", "search")
    .boost(2.0)
    .build()
    .unwrap();

let content_query = QueryBuilder::new()
    .term("content", "engine")
    .build()
    .unwrap();

let combined_query = QueryBuilder::and(vec![title_query, content_query])
    .build()
    .unwrap();

Python API​

QueryNode and Value Classes​

The Python API provides equivalent classes for constructing query nodes:

Important Note about Range Queries: When creating range queries, always use the direct field name (e.g., "word_count") and not the nested path (e.g., "metadata.word_count"), even though the field may appear nested in document structure. Using nested paths with range queries will not return the expected results.

from fast_inverted_index import PyQueryNode, PyValue, PyQueryBound

# Create a simple term query
query = PyQueryNode.term("content", "search")

# Create a boosted term query
query = PyQueryNode.term("title", "search", boost=2.0)

# Create a phrase query
query = PyQueryNode.phrase("content", ["fast", "search"], 2)

# Create a prefix query
query = PyQueryNode.prefix("content", "prog")

# Create a fuzzy query
query = PyQueryNode.fuzzy("content", "search", 1)

# Create a range query
start_bound = PyQueryBound.included(PyValue.integer(100))
end_bound = PyQueryBound.excluded(PyValue.integer(500))
# IMPORTANT: Use direct field name, not a nested path like "metadata.word_count"
query = PyQueryNode.range("word_count", start_bound, end_bound)

# Combine with boolean operators
query = PyQueryNode.and([
    PyQueryNode.term("content", "search"),
    PyQueryNode.term("title", "engine")
])

# NOT query
query = PyQueryNode.not(PyQueryNode.term("content", "slow"))

# Filter query
query = PyQueryNode.filter(
    PyQueryNode.term("content", "search"),
    PyQueryNode.term("category", "technology")
)

Query Execution​

The Python API exposes methods for executing queries:

from fast_inverted_index import Index, PyQueryNode, PyQueryExecutionParams

# Create an index
index = Index(in_memory=True)

# Add some documents
index.add_document(1, "This is a test document about search engines")
index.add_document(2, "Fast search is important for good user experience")

# Create a query
query = PyQueryNode.term("content", "search")

# Execute the query with default parameters
result = index.execute_query(query)

# Print the results
for doc_id, score in result.scored_docs:
    print(f"Document {doc_id}: score {score}")

# Execute with custom parameters
params = PyQueryExecutionParams(
    scoring_algorithm="bm25l",
    explain=True,
    limit=10
)

result = index.execute_query(query, params)

# Get explanations
explanations = result.explanations
if explanations:
    for doc_id, explanation in explanations.items():
        print(f"Explanation for document {doc_id}:")
        print(explanation.to_tree())

Convenience Methods​

The Index class provides convenience methods for common query patterns:

# Search in a specific field
results = index.search_field("title", "search", algorithm="bm25", limit=10)

# Search for a phrase
results = index.search_phrase("content", ["fast", "search"], slop=2, 
                             algorithm="bm25", limit=10)

# Search multiple fields with different boosts
field_values = [
    ("title", "search", 2.0),
    ("content", "engine", 1.0),
    ("tags", "search", 1.5)
]
results = index.search_multi_field(field_values, algorithm="bm25", limit=10)

Advanced Examples​

Complex Query with Field Boosting​

from fast_inverted_index import Index, PyQueryNode, PyQueryExecutionParams

# Create an index
index = Index(in_memory=True)

# Add some documents with metadata
index.add_document(
    doc_id=1,
    content="Fast search engines use inverted indexes for efficient retrieval",
    metadata={
        "title": "Understanding Search Engines",
        "tags": ["search", "indexing", "performance"],
        "category": "Information Retrieval",
    }
)

index.add_document(
    doc_id=2,
    content="Inverted indexes map terms to document IDs for fast lookups",
    metadata={
        "title": "Inverted Indexes Explained",
        "tags": ["indexing", "data structures", "algorithms"],
        "category": "Information Retrieval",
    }
)

# Create a complex query
title_query = PyQueryNode.term("title", "search", boost=2.0)
content_query = PyQueryNode.or([
    PyQueryNode.term("content", "index"),
    PyQueryNode.term("content", "search")
])
category_filter = PyQueryNode.term("category", "Information Retrieval")

# Combine queries with filter
query = PyQueryNode.filter(
    PyQueryNode.or([title_query, content_query]),
    category_filter
)

# Create custom execution parameters
params = PyQueryExecutionParams(
    scoring_algorithm="bm25",
    explain=True,
    limit=10,
    field_boosts={"title": 1.5, "tags": 1.2}
)

# Execute the query
result = index.execute_query(query, params)

# Process results
print(f"Found {result.len()} results in {result.time_ms:.2f}ms")
for doc_id, score in result.scored_docs:
    doc = index.get_document(doc_id)
    print(f"- {doc['title']} (Score: {score:.4f})")
    
    # Show explanation if available
    if result.explanations and doc_id in result.explanations:
        explanation = result.explanations[doc_id]
        print(f"  Explanation: {explanation.description}")
        print(f"  Score components: {len(explanation.details)} factors")

Date Range Query​

import time
from fast_inverted_index import Index, PyQueryNode, PyValue, PyQueryBound

# Create an index
index = Index(in_memory=True)

# Add documents with timestamps
current_time = int(time.time())
one_day = 24 * 60 * 60

for i in range(10):
    timestamp = current_time - (i * one_day)  # Each doc is one day older
    index.add_document(
        doc_id=i+1,
        content=f"Document created on day {i}",
        metadata={
            "title": f"Document {i+1}",
            "created_at": timestamp
        }
    )

# Create a range query for documents from the last 3 days
three_days_ago = current_time - (3 * one_day)
start_bound = PyQueryBound.included(PyValue.integer(three_days_ago))
end_bound = PyQueryBound.unbounded()

# Note: Always use the direct field name, not a nested path
date_query = PyQueryNode.range("created_at", start_bound, end_bound)

# Execute the query
result = index.execute_query(date_query)

# Show the results
print(f"Documents from the last 3 days: {result.len()}")
for doc_id, score in result.scored_docs:
    doc = index.get_document(doc_id)
    created_days_ago = (current_time - doc["created_at"]) / one_day
    print(f"- {doc['title']} (created {created_days_ago:.1f} days ago)")

Combining Text Search with Filtering​

from fast_inverted_index import Index, PyQueryNode, PyQueryExecutionParams

# Create an index
index = Index(in_memory=True)

# Add some documents with various metadata
for i in range(1, 11):
    # Even IDs are programming docs, odd are cooking docs
    category = "Programming" if i % 2 == 0 else "Cooking"
    
    # Word counts vary by document
    word_count = 100 + (i * 50)
    
    content = ""
    if category == "Programming":
        content = "Python is a versatile programming language for many applications."
    else:
        content = "Italian pasta recipes are popular in modern cooking."
    
    # Add the document with metadata
    index.add_document(
        doc_id=i,
        content=content,
        metadata={
            "title": f"{category} Document {i}",
            "category": category,
            "metadata": {"word_count": word_count}
        }
    )

# Create a query that finds programming documents with "python" 
# and at least 300 words
text_query = PyQueryNode.term("content", "python")

# Word count >= 300
word_count_start = PyQueryBound.included(PyValue.integer(300))
word_count_end = PyQueryBound.unbounded()
# Note: Use direct field name "word_count", not "metadata.word_count" 
word_count_query = PyQueryNode.range("word_count", word_count_start, word_count_end)

# Category filter
category_query = PyQueryNode.term("category", "Programming")

# Combine them all
combined_query = PyQueryNode.and([
    text_query,
    PyQueryNode.filter(word_count_query, category_query)
])

# Execute the query
result = index.execute_query(combined_query)

# Show the results
print(f"Found {result.len()} programming documents about Python with 300+ words")
for doc_id, _ in result.scored_docs:
    doc = index.get_document(doc_id)
    print(f"- {doc['title']} (Words: {doc['metadata']['word_count']})")

Performance Considerations​

  1. Query Complexity: Very complex queries with many nodes can increase execution time. Consider simplifying queries where possible.

  2. Field Boosts: Using many field boosts increases scoring computation time. Focus on the most important fields.

  3. Explanation Generation: Enabling the explain parameter adds overhead. Use it for debugging but disable in production.

  4. Scoring Algorithm Selection:

    • TF-IDF: Fastest, but less sophisticated
    • BM25: Good balance of speed and quality
    • BM25L: Better for varied document lengths, slightly slower
    • LearningToRank: Most powerful but most computationally expensive

  5. Caching: The query engine benefits from the index's internal caching. Configure cache parameters appropriately for your workload.

For more information on performance optimization, see the Performance documentation.

BM25 and BM25L Scoring​

The Query Engine implements both BM25 and BM25L ranking algorithms to provide state-of-the-art relevance scoring.

Non-Negative Scoring Guarantee​

Both BM25 and BM25L algorithms in this library are implemented with a non-negative score guarantee. This means:

  1. All document scores will be ≥ 0.0, even for terms that appear in most documents
  2. The IDF component is guaranteed to be non-negative, preventing negative contribution from common terms
  3. Term scores and final document scores have a floor of 0.0

This approach helps ensure consistent, intuitive ranking behavior, especially when:

  • A term appears in nearly all documents in the collection
  • Working with small document collections
  • Using boolean combinations of term queries

Zero Scores for High-Frequency Terms​

It's important to note that our implementation will produce scores of 0.0 for terms that appear in a large proportion of documents (typically more than 50-55% of the corpus). This is expected behavior after clamping negative IDF values to zero.

Through extensive testing with synthetic and real data, we've found:

  1. When a term's document frequency exceeds approximately 53.5% of the corpus, the raw IDF value becomes negative
  2. After applying our non-negative guarantee, these terms receive scores of 0.0
  3. This behavior is consistent across both BM25 and BM25L algorithms

This approach aligns with information retrieval best practices, as extremely common terms provide little discriminative value for ranking.

For a detailed explanation and test results, see the BM25L documentation.

Field-Specific Searching​

The Query Engine provides comprehensive support for field-specific searching, allowing you to direct queries to particular fields and control relevance through boosting.

Rust API for Field Searching​

use fast_inverted_index::query::{QueryNode, QueryExecutionParams};
use std::collections::HashMap;

// Search in a specific field
let title_query = QueryNode::term("title", "search");

// Multi-field search with different boosts
let multi_field_query = QueryNode::or(vec![
    QueryNode::term_with_boost("title", "search", 5.0),  // Title is 5x more important
    QueryNode::term_with_boost("content", "search", 1.0),  // Content has normal importance
    QueryNode::term_with_boost("tags", "search", 3.0),  // Tags are 3x more important
]);

// Execute the query with additional field boosts in the parameters
let mut params = QueryExecutionParams::default();
let mut field_boosts = HashMap::new();
field_boosts.insert("title".to_string(), 2.0);  // Additional 2x boost for title
field_boosts.insert("tags".to_string(), 1.5);   // Additional 1.5x boost for tags
params.field_boosts = field_boosts;

let results = index.execute_query(&multi_field_query, Some(params))?;

Python API for Field Searching​

from fast_inverted_index import QueryNode, QueryExecutionParams

# Convenience method for field-specific search
title_results = index.search_field("title", "search", ranking_method="bm25", limit=10)

# Multi-field search with different boosts using QueryNode API
multi_field_query = QueryNode.OR([
    QueryNode.term("title", "search", boost=5.0),   # Title is 5x more important
    QueryNode.term("content", "search", boost=1.0),  # Content has normal importance
    QueryNode.term("tags", "search", boost=3.0),    # Tags are 3x more important
])

# Execute with additional field boosts in the parameters
params = QueryExecutionParams(
    scoring_algorithm="bm25",
    limit=10,
    field_boosts={
        "title": 2.0,  # Additional 2x boost for title
        "tags": 1.5    # Additional 1.5x boost for tags
    }
)

results = index.execute_query(multi_field_query, params)

# Simple multi-field search with the convenience API
boost_fields = {
    "title": 5.0,    # Title is 5x more important
    "content": 1.0,  # Content has normal importance
    "tags": 3.0,     # Tags are 3x more important
}
simple_results = index.search("search", ranking_method="bm25", boost_fields=boost_fields)

Combined Boosts Calculation​

When using both per-query boosts and global field boosts, the final boost value is calculated as:

final_boost = query_boost * field_boost

For example, if you set:

  • A query boost of 5.0 on the title field in the query node
  • A global field boost of 2.0 in the execution parameters

The final boost for the title field would be 10.0 (5.0 * 2.0).

Field Score Normalization​

The Query Engine ensures that field scores are properly normalized so that:

  1. Matches across all fields contribute to the final score
  2. Even small contributions from secondary fields are preserved
  3. Zero scores are avoided for fields that match the query

This ensures that documents matching in multiple fields are correctly ranked higher than those matching in only one field, even when field boosts create large differences in raw score values.

Best Practices​

  1. Use Field-Specific Searching: Direct queries to specific fields when possible to improve relevance.

  2. Combine Text Search with Filters: Use filter nodes for non-scoring constraints to improve performance.

  3. Boost Important Fields: Use field boosting to prioritize matches in important fields like titles.

  4. Prefer Term and Phrase Queries: These are the most efficient. Use prefix and fuzzy queries sparingly.

  5. Use Direct Field Names for Range Queries: For range queries, always use the direct field name (e.g., "word_count") and not the nested path (e.g., "metadata.word_count") even though the field may appear nested in document structure.

  6. Debug with Explanations: When relevance is unexpected, use the explanation feature to understand scoring.

  7. Choose the Right Scoring Algorithm: Match the algorithm to your content and search requirements.

  8. Combine Query and Parameter Boosts Strategically: Use query-specific boosts for term-level importance and field boosts in parameters for global field importance.

  9. Use the Schema System: Define a proper schema with field types and default boosts for consistent search behavior.
