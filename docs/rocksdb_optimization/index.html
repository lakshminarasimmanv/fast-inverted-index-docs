RocksDB Optimization Guide

This guide covers the optimized RocksDB storage backend added in version 0.4.7 of Fast Inverted Index. The optimizations significantly improve indexing and query performance when using persistent storage.

Overview​

Fast Inverted Index uses RocksDB as its persistent storage backend. RocksDB is a high-performance embedded key-value store designed for SSD storage. Our optimized implementation provides significant performance improvements, especially for bulk indexing operations.

Key optimizations include:

  1. WriteBatch Support - Consolidate multiple operations into atomic batches
  2. Bulk Loading Mode - Disable WAL for maximum performance during initial loading
  3. Column Family Tuning - Optimized settings for different data types
  4. Thread-Safe Wrappers - Robust multi-threading support for Python bindings
  5. Performance Metrics - Detailed instrumentation for monitoring
  6. Batch Operations - High-level batch APIs for document operations

Configuration Options​

The RocksDbConfig struct provides comprehensive options for tuning RocksDB performance:

pub struct RocksDbConfig {
    // Base path for the database
    pub path: PathBuf,
    
    // Whether to create a new database if one doesn't exist
    pub create_if_missing: bool,
    
    // Maximum number of open files
    pub max_open_files: i32,
    
    // Write buffer size in bytes
    pub write_buffer_size: usize,
    
    // Maximum number of write buffers
    pub max_write_buffer_number: i32,
    
    // Target file size in bytes
    pub target_file_size_base: u64,
    
    // Whether to disable WAL during bulk loading
    pub disable_wal_for_bulk: bool,
    
    // Whether to use write batching
    pub use_write_batch: bool,
    
    // Batch size for write operations
    pub write_batch_size: usize,
    
    // Maximum background jobs for compaction
    pub max_background_jobs: i32,
    
    // Whether to enable statistics
    pub enable_statistics: bool,
    
    // Whether to optimize for bulk loading
    pub optimize_for_bulk: bool,
    
    // Interval to flush batches even if not full
    pub batch_flush_interval: Option<Duration>,
}

Optimizing for Different Workloads​

Bulk Loading​

For initial data loading or batch imports, optimize for maximum write throughput:

let config = RocksDbConfig {
    path: "/path/to/index".into(),
    
    // Increase write buffer for better batching
    write_buffer_size: 128 * 1024 * 1024, // 128 MB
    max_write_buffer_number: 6,
    
    // Enable bulk loading optimizations
    disable_wal_for_bulk: true,
    use_write_batch: true,
    write_batch_size: 10000, // Large batch size
    optimize_for_bulk: true,
    
    // More background jobs for better parallelism
    max_background_jobs: 8,
    
    // Other settings at defaults
    ..Default::default()
};

Read-Heavy Workloads​

For query-intensive applications, optimize for read performance:

let config = RocksDbConfig {
    path: "/path/to/index".into(),
    
    // More file descriptors for better read performance
    max_open_files: 2000,
    
    // Keep write buffer reasonable
    write_buffer_size: 32 * 1024 * 1024, // 32 MB
    
    // Enable batching but with smaller batches
    use_write_batch: true,
    write_batch_size: 100,
    
    // No WAL disabling for safety
    disable_wal_for_bulk: false,
    
    // Other settings at defaults
    ..Default::default()
};

Balanced Approach​

For balanced performance with both reads and writes:

let config = RocksDbConfig {
    path: "/path/to/index".into(),
    
    // Moderate file descriptor count
    max_open_files: 1000,
    
    // Moderate write buffer
    write_buffer_size: 64 * 1024 * 1024, // 64 MB
    max_write_buffer_number: 4,
    
    // Enable batching with medium batch size
    use_write_batch: true,
    write_batch_size: 1000,
    
    // Enable WAL for safety
    disable_wal_for_bulk: false,
    
    // Other settings at defaults
    ..Default::default()
};

Using Bulk Loading Mode​

The bulk loading mode is particularly useful for initial data loading or large batch operations. It significantly improves write performance by disabling the Write-Ahead Log (WAL) and applying other optimizations.

// Create store with configuration
let mut store = RocksDbStore::with_config(config)?;

// Enable bulk loading mode before large indexing operation
store.set_bulk_loading_mode(true);

// Perform your bulk indexing operations...
// This will be significantly faster

// Disable bulk loading mode when finished (important for durability)
store.set_bulk_loading_mode(false);

Important: When bulk loading mode is enabled with WAL disabled, there is a risk of data loss if the process crashes during indexing. Always disable bulk loading mode and perform a backup when the operation completes.

Column Family Optimization​

Different column families are optimized for their specific data and access patterns:

  1. CF_DOCS (document metadata) - Optimized for random access with LZ4 compression
  2. CF_TERMS (term dictionary) - Configured with prefix bloom filters for efficient lookups
  3. CF_POSTINGS (posting lists) - Optimized for sequential writes and reads
  4. CF_TERM_STATS (term statistics) - Standard configuration for small key-value pairs
  5. CF_METADATA (index metadata) - Standard configuration for occasional access

Performance Monitoring​

The storage layer collects detailed metrics to help identify bottlenecks:

// Get statistics summary
let stats_summary = store.metrics.summary();
println!("{}", stats_summary);

// Access individual metrics
let write_count = store.metrics.write_count.load(Ordering::Relaxed);
let avg_batch_size = store.metrics.avg_items_per_batch();
let avg_write_time = store.metrics.avg_write_time_us();

Available metrics include:

  • Write counts and timing
  • Read counts and timing
  • Batch operation statistics
  • WAL flush metrics
  • Compaction metrics
  • Column family operation counts

Python Integration​

The optimized RocksDB implementation is fully available through the Python bindings:

from fast_inverted_index import Index, Schema, FieldSchema

# Create schema
schema = Schema()
schema.add_field(FieldSchema.text("content"))

# Create index with RocksDB storage and optimized config
index = Index(
    storage_path="/path/to/index",
    in_memory=False,  # Use RocksDB storage
    schema=schema
)

# The Python bindings automatically use batched operations
# when adding multiple documents

# Add documents in batch for better performance
documents = [
    (1, {"content": "Document 1 content"}),
    (2, {"content": "Document 2 content"}),
    # ...more documents
]

# This will use the optimized batch operations internally
index.add_documents_with_fields_parallel(documents)

Best Practices​

  1. Use Batched Operations - Always prefer batch operations over individual document operations when possible.

  2. Tune for Your Hardware - Adjust settings based on available RAM, CPU cores, and storage type:

    • SSDs: Larger write buffers, more background jobs
    • HDDs: Smaller, more frequent compactions, smaller batches

  3. Monitor Performance - Track metrics to identify bottlenecks and adjust configurations accordingly.

  4. Backup Regularly - Create periodic backups, especially after bulk loading operations.

  5. Scale the Write Buffer - For machines with more RAM, increase the write buffer size for better performance.

Troubleshooting​

  1. High Write Latency

    • Check if batching is enabled (use_write_batch: true)
    • Consider increasing write buffer size
    • Enable bulk loading mode for initial imports

  2. High Memory Usage

    • Reduce write buffer size and number
    • Reduce batch size
    • Consider tuning bloom filter sizes

  3. Slow Reads

    • Ensure max_open_files is set appropriately
    • Check if bloom filters are properly configured
    • Verify proper column family settings

  4. Database Size Issues

    • Run periodic compactions
    • Consider adjusting compression settings
    • Check if WAL is properly flushed

Conclusion​

The optimized RocksDB storage backend significantly improves the performance of Fast Inverted Index, especially for bulk operations. By properly configuring the RocksDB settings for your specific workload, you can achieve optimal performance for both indexing and querying operations.

For more information about general performance tuning, see the Performance Tuning Guide.
